#1 High Impact Fix Now
Legacy duplicate collection coexists with Primitive Colours system
Why it matters: Designers unknowingly pick legacy MCP names instead of the correct Primitive Colours names (e.g., DeepSeaBlue/500, Yellow/500, Neutral/950). Components break when one system updates but the other does not. Dev handoff becomes ambiguous with two valid sources for the same colour.
Recommended Fix
Archive or delete the entire legacy collection (MCP-sourced names like color/azure/*, color/grey/*, item spacing/*, font family/*, font weight/*, font size/*, line height/*, radius/*, stroke weight/*). Remap all usages to the correct Primitive Colours (e.g., DeepSeaBlue/500, Greyscale/100), _spacing (e.g., space-12, space-20), and Typography (e.g., font/family/body, font/weight/bold) collections.

#2 High Impact Fix Now
Neutral/100 and Neutral/200 resolve to the same hex (#FDFCF9)
Why it matters: Two primitive steps that are visually identical waste a slot in the scale and make the semantic token greyscale/surface/subtle (aliased to Neutral/100) indistinguishable from greyscale/surface/default (Neutral/200). Designers lose the ability to create visual layering.
Recommended Fix
Change Neutral/100 to a distinct warm off-white such as #FAF8F3 so that surface/subtle is perceptibly lighter than surface/default. Alternatively, re-alias greyscale/surface/subtle to Neutral/300 (#FBF9F4).

#4 High Impact Fix Now
No hover, focus, or active surface/border semantic tokens exist
Why it matters: Every interactive element (button, link, card, input, dropdown item) needs visible state changes for usability and accessibility. Without these tokens, developers will hardcode arbitrary colours creating inconsistency. WCAG 2.1 SC 2.4.7 requires visible focus indicators.

Add four tokens to the Colour Tokens collection: greyscale/surface/hover, greyscale/surface/active, greyscale/border/focus, greyscale/border/hover.

6 Business & UX Impact

The real cost of the top 5 issues, measured in conversions, development time, and business risk.

⚠

Missing Hover/Focus Tokens — #4

👤 Conversion / Usability

Users cannot see what is clickable or which element has keyboard focus. E-commerce sites with poor interactive feedback see measurably higher cart abandonment because users lose trust in the interface.

🔧 Development Time

Every developer creates their own hover colour, leading to 5-10 different greys across the product. Rework to unify later costs 2-3x the effort of doing it right now.

💰 Cost of NOT Fixing

Accessibility lawsuit risk (WCAG 2.1 SC 2.4.7 violation). Remediation costs post-launch are typically 5-10x higher than prevention during the design phase.

🛠

Legacy Duplicate Collection — #1

👤 Conversion / Usability

When colours drift between legacy MCP-sourced names and the correct Primitive Colours (e.g., DeepSeaBlue/500, Yellow/500), buttons and badges become inconsistent. Users subconsciously distrust interfaces with visual inconsistency, reducing conversion.

🔧 Development Time

Developers waste 15-30 minutes per component deciding which colour source to use. Over 20+ components, this compounds to full days of wasted decision-making.

💰 Cost of NOT Fixing

Every week the duplicate exists, more components get built on the wrong source. Migration cost grows linearly. Deleting now: 1 hour. Deleting in 3 months: 1-2 full days.

■

No Shadow/Elevation Tokens — #6

👤 Conversion / Usability

Cards, modals, and dropdowns without systematic depth feel flat and disorganized. Users struggle to distinguish foreground actions from background content, reducing task completion rates.

🔧 Development Time

Each developer invents box-shadow values. Code review catches inconsistencies, spawning back-and-forth. Token-driven shadows eliminate this entirely.

💰 Cost of NOT Fixing

The perceived quality gap between "polished" and "amateur" e-commerce is often just elevation consistency. Competitors with systematic depth hierarchy will appear more trustworthy.

⚫

Identical Neutral/100 & Neutral/200 — #2

👤 Conversion / Usability

Visual layering (page > card > element) depends on subtle surface steps. When two stops are identical, cards blend into pages and content hierarchy collapses, making the layout feel flat.

🔧 Development Time

Developers will manually override colours to create visual separation that the token system should provide. These overrides accumulate as tech debt.

💰 Cost of NOT Fixing

A 2-minute primitive value change now prevents dozens of manual overrides across the entire component library.

⚠

Missing Warning Mode — #9

👤 Conversion / Usability

Low-stock indicators, price-drop alerts, and partial-form warnings all need a distinct "warning" visual that is not "error". Conflating warning with error causes users to panic or ignore real errors.

🔧 Development Time

Without a Warning mode, developers hardcode yellow values or misuse the Error mode. Both approaches require rework when the Warning mode is eventually added.

💰 Cost of NOT Fixing

Warning patterns exist in almost every e-commerce flow (checkout, inventory, forms). Not having this mode from the start means retrofitting 10-15 components later.
